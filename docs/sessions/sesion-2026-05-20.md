## 3. DECISIONES TÉCNICAS TOMADAS

### Decisión 1: Backend usa `service_role` key para Storage
- **Qué**: El cliente de Supabase en el backend usa `SUPABASE_SERVICE_ROLE_KEY` en vez de `SUPABASE_KEY` (anon)
- **Por qué**: Las operaciones de Storage (`create_signed_upload_url`) requieren bypass de RLS. El backend es un servidor confiable.
- **Tradeoff**: La key anon queda disponible para el frontend solo. El backend tiene acceso total.

### Decisión 2: `get_parametros_by_variedad` con fallback
- **Qué**: Si no hay parámetros configurados en `plantilla_parametro` para la variedad, devuelve todos los parámetros activos
- **Por qué**: Evita que el frontend se rompa si una variedad no tiene configuración de plantilla
- **Tradeoff**: El fallback puede devolver parámetros irrelevantes, pero es mejor que un array vacío

### Decisión 3: Validación de ficha — check existente primero
- **Qué**: `generar_ficha` verifica si ya existe ficha activa **antes** de validar requisitos
- **Por qué**: Permite que un lote en `listo_para_compartir` pueda consultar su ficha existente sin error
- **Tradeoff**: Ninguno — es el comportamiento correcto

### Decisión 4: `producto_principal` excluido de parámetros por variedad
- **Qué**: El parámetro ID 46 (producto_principal) se excluye explícitamente del endpoint `/variedades/{id}/parametros`
- **Por qué**: Se auto-agrega automáticamente cuando la suma de valores < peso de muestra
- **Tradeoff**: El frontend no debe enviarlo manualmente
