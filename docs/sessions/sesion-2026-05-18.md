## 4. DECISIONES TÉCNICAS TOMADAS

### Decisión 1: Conexión directa a DB en vez de pgbouncer
- **Qué**: `DATABASE_URL` apunta a `aws-1-us-east-1.pooler.supabase.com:5432` en vez de `:6543`
- **Por qué**: pgbouncer en modo transaction no soporta prepared statements con asyncpg. SQLAlchemy ya maneja el pool de conexiones correctamente.
- **Tradeoff**: Mayor consumo de conexiones en DB, pero mejor performance y compatibilidad.

### Decisión 2: JWKS cacheado al startup
- **Qué**: JWKS se fetchea una vez al iniciar el servidor y se guarda en memoria
- **Por qué**: Elimina HTTP calls a Supabase Auth por cada request. El JWKS rara vez cambia.
- **Tradeoff**: Si Supabase rota las keys, hay que reiniciar el servidor.

### Decisión 3: Cache TTL en memoria
- **Qué**: Cache simple con TTL (5 min user/org, 1 min lote)
- **Por qué**: Reduce latencia de ~500ms a <10ms para requests repetidos. No requiere infraestructura adicional (Redis).
- **Tradeoff**: Cache no distribuido (problema si hay múltiples instancias). Para producción con escalado horizontal, migrar a Redis.

### Decisión 4: pool_use_lifo=True
- **Qué**: El pool reutiliza la conexión más reciente en vez de la más antigua
- **Por qué**: Mejora la localidad de caché y reduce latencia en conexiones reutilizadas.
- **Tradeoff**: Ninguno significativo para este caso de uso.

### Decisión 5: Eliminar view_count UPDATE del GET
- **Qué**: El endpoint GET ya no incrementa `view_count`
- **Por qué**: Una lectura no debería tener efectos secundarios de escritura. Simplifica el flujo y elimina un roundtrip.
- **Tradeoff**: `view_count` ya no se actualiza automáticamente. Si se necesita, implementar como endpoint separado o background job.

## 5. PROBLEMAS ENCONTRADOS Y SOLUCIONES

### Problema 1: Import circular (api.py → dependencies.py → api.py)
- **Error**: `ImportError: cannot import name '_jwks_cache' from partially initialized module`
- **Causa**: `dependencies.py` importaba `_jwks_cache` de `api.py`, pero `api.py` importaba de `dependencies.py` vía `router.py`
- **Solución**: Mover `_jwks_cache` a `src/core/supabase_client.py` (sin dependencias circulares)

### Problema 2: JWT alg no permitido
- **Error**: `The specified alg value is not allowed`
- **Causa**: Hardcodeamos `algorithms=["HS256"]` pero Supabase usa RS256
- **Solución**: Leer `alg` dinámicamente del header del token con `jwt.get_unverified_header(token)`

### Problema 3: PEM file malformed
- **Error**: `Unable to load PEM file. MalformedFraming`
- **Causa**: Supabase usa RS256 (asymmetric), necesitaba la clave pública (JWKS), no un secret simétrico
- **Solución**: Implementar fetch de JWKS al startup y usar la key correspondiente al `kid` del token

### Problema 4: Prepared statements con pgbouncer
- **Error**: `prepared statement "__asyncpg_stmt_1__" already exists`
- **Causa**: pgbouncer en modo transaction (puerto 6543) no soporta prepared statements
- **Solución**: Usar conexión directa (puerto 5432) y eliminar `statement_cache_size=0`

### Problema 5: DNS no resuelve db.hregnoaxqsrqjhosvwzx.supabase.co
- **Error**: `No address associated with hostname`
- **Causa**: El hostname `db.*.supabase.co` no es accesible desde el container Docker
- **Solución**: Usar `aws-1-us-east-1.pooler.supabase.com:5432` (el pooler acepta conexiones en puerto 5432 también)

### Problema 6: Primera request tarda 9 segundos
- **Causa**: Latencia de red container Docker (local) → AWS US-East-1 (Virginia) + SSL handshake + auth
- **Solución**: Inevitable con la arquitectura actual. Se mitiga con:
  - Connection pool (requests posteriores reutilizan conexión)
  - Cache TTL (requests repetidos no van a DB)
  - Para producción: deployar en la misma región que Supabase

### Problema 7: Seq Scan en tablas pequeñas
- **Observación**: PostgreSQL usa Seq Scan en vez de índices para tablas < 10 filas
- **Causa**: El planner decide que scan secuencial es más rápido para tablas tiny
- **Solución**: Correcto y óptimo. Los índices se usarán automáticamente cuando las tablas crezcan.
