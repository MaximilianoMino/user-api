## 3. DECISIONES TÉCNICAS TOMADAS

### Decisión 1: Diccionario `MAPEO_GEMINI_DB` + fallback ilike
- **Qué**: Se busca primero en el diccionario por clave exacta, luego fallback con `ilike` en Python
- **Por qué**: El diccionario da matches precisos, ilike cubre casos no mapeados sin consultas SQL
- **Tradeoff**: El diccionario hay que mantenerlo manualmente

### Decisión 2: Batch query para parámetros
- **Qué**: Una sola query trae todos los `(parametro_id, codigo)` y se buscan en memoria
- **Por qué**: Pasa de ~60 queries a 1 por escaneo
- **Tradeoff**: Más memoria, pero solo ~50 filas, es despreciable

### Decisión 3: Content negotiation en ficha pública
- **Qué**: Mismo endpoint sirve HTML (navegador) o JSON (app mobile) según `Accept` header
- **Por qué**: Evita duplicar endpoints y mantiene compatibilidad con la app
- **Tradeoff**: Lógica de renderizado en el backend (no reutilizable desde frontend SPA)

### Decisión 4: Relación directa `FichaReporte.lote` en vez de navegar por `muestra.lote`
- **Qué**: `FichaReporte` ya tiene FK directa a `lote`. Se usa esa en vez de `muestra.lote`
- **Por qué**: Evita el selectinload encadenado que fallaba
- **Tradeoff**: Ninguno — la FK directa es más eficiente
